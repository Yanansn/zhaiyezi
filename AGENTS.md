# 摘叶子 Agent 工作规则

## 项目目标

通过处理真实开源 Issue，在理解项目原理和社区协作方式的同时，完成可验证的代码贡献，并保存可查询、可恢复的全过程记录。

## 启动与恢复

每次开始工作或更换上下文时，必须按顺序执行：

1. 使用 `harvest-open-source-issue` Skill。
2. 读取根目录 `HANDOFF.md`。
3. 读取 `registry/issues.yaml`，确认当前活动 Issue。
4. 读取活动 Issue 的 `STATUS.yaml` 和 `JOURNAL.md`。
5. 按需读取 `ANALYSIS.md`、`CODE-MAP.md`、`PLAN.md`、`IMPLEMENTATION.md`、`TESTING.md`、`LEARNING.md` 和 `PR.md`。
6. 从 GitHub 核验 Issue、PR、评论、标签、认领人和 CI 的实时状态。
7. 检查当前 Git 分支、最近提交和未提交改动。
8. 在继续执行前，向用户报告“记录状态、实时状态、差异、阻塞和下一步”。

项目文件是历史决定与工作状态的权威记录；GitHub 是外部 Issue、PR 和 CI 实时状态的权威来源。不得仅依靠对话记忆继续任务。

## 标准工作流

- 先筛选，再解释，再建立代码地图，然后设计、实现、测试和贡献。
- 区分来源事实、代码事实、测试证据和推断。
- 在编码前解释受影响子系统、当前行为、预期行为、根因或待验证假设。
- 修改必须聚焦当前 Issue，不混入无关重构。
- 测试必须记录工作目录、前置条件、命令、目的、结果和限制。
- 每次重要状态变化都更新 `STATUS.yaml`、`JOURNAL.md` 和根目录 `HANDOFF.md`。
- Issue 完成或终止时，必须记录最终结果、原因、Review 反馈和学习总结。

## 教学要求

- 不把用户当作只复制命令的执行者。
- 解释重要代码为什么存在、修改为什么有效、测试为什么能够证明结果。
- 新术语首次出现时用易懂语言解释，并说明它在项目调用链中的位置。
- 给出命令时说明运行位置、预期输出和常见失败原因。

## 外部操作审批

以下操作必须在执行前取得用户明确确认：

- 公开评论或回复 Issue、PR
- 认领或分配 Issue
- 推送分支到外部仓库
- 创建、关闭或合并 PR
- 请求 Reviewer 或代表用户作出社区承诺

只读调查、本地代码修改、与任务直接相关的本地测试和记录更新可以自动执行。

## 子 Agent

仅将独立、边界清晰、以读取和复核为主的工作委派给子 Agent，例如历史调查、代码路径探索、测试缺口或独立 Review。主 Agent 负责合并结论并更新事实库。同一代码区域避免多个 Agent 并行写入。

## 完成条件

只有当状态、实现、测试证据、PR/终止结果和学习总结均已记录时，一个 Issue 才算闭环。
