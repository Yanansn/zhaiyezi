# DECISION-0002：普通 Chat 与本地 Codex 分工

## 决定

《摘叶子》采用“普通 Chat 做老师和方案顾问，本地 Codex 做阶段化工程执行，公开 GitHub 仓库做持久事实通道”的运行模型。

普通 Chat 负责候选筛选、原理讲解、学习路径、方案比较、维护者评论解读和生成 `Execution Brief`。Codex 只在收到简报后执行一个明确阶段：读取代码、建立代码路径、实现、测试、诊断、更新事实记录，以及执行已授权的 Git/GitHub 操作。

本决定取代 DECISION-0001 中“主 Agent 负责调查、推理、实现和推进”的宽泛职责；其事实库和外部审批原则继续有效。

## 原因

- 长篇教学和方案讨论不需要本地终端，放在普通 Chat 更经济。
- Codex 用量应集中在真实代码、测试、CI 和 Git 操作上。
- 每个阶段只复制一次紧凑简报，减少重复上下文和搜索。
- Codex 将授权范围内的结果推送到公开事实仓库后，普通 Chat 可通过 GitHub Connector 读取，不需要人工复制全部日志。

## 执行约束

- 没有 `Execution Brief` 时，Codex 不开展宽泛筛选、讲解或编码。
- 每份简报只定义一个阶段，并列出交付物和审批边界。
- 默认不使用子 Agent。
- 机械搜索、格式检查、简单测试和状态更新优先使用较轻模型；复杂根因、CI 和 Review 再使用强模型。
- 不重复普通 Chat 已确认的知识，只报告工程变更、理由、验证和风险。
- 记录按发生变化的事实增量更新，不在每轮重写全部 Issue 文件。
- Commit、Push、PR、公开评论和其他社区动作必须在简报中明确授权，或另行获得确认。
